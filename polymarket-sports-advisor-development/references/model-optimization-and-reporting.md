# Model Optimization and Report UX Notes

Use this reference when extending a read-only Polymarket sports advisor beyond a simple Elo/Poisson baseline.

## Model Layers

Keep the model explainable and layered rather than turning it into an opaque feature pile:

1. **Base strength**: Elo ratings feed expected goals.
2. **Team form**: derive `attack_multiplier` and `defense_multiplier` from recent historical goals for/against relative to the global average. Clamp outputs and require a minimum match count. In this codebase lower `defense_multiplier` means stronger defense.
3. **Manual context**: apply injury, lineup, travel, motivation, or tactical adjustments after team form. Keep notes auditable and show them in reports.
4. **Score model**: use Poisson with optional Dixon-Coles `rho` for low-score dependence. Keep `rho=0.0` as the independent-Poisson fallback.
5. **Market comparison**: compute `edge = model_probability - market_buy_price`; optional market shrinkage uses `final_prob = model_weight * model_prob + (1 - model_weight) * market_buy_price`.
6. **Simulation**: Monte Carlo can summarize selected known matches, but do not call it full bracket/champion odds unless knockout fixtures and advancement logic are actually implemented.

## P1 Split Pitfall

When asked for P1 model improvements, treat it as two separate deliverables:

- data-derived attack/defense form multipliers (`team_form`)
- manual injury/lineup/context adjustments (`team_adjustments`)

Doing only one is incomplete.

## Scanner Report UX

For beginner-friendly match scanner reports:

- Default report window should be the next 24 hours unless the user asks otherwise.
- Sort cards by kickoff time from soonest to latest; do not reorder by BUY/edge because it breaks the match-day timeline.
- Show kickoff time in the user's expected timezone; for this World Cup workflow use Beijing time.
- Report every scheduled match in the window, including `NO MARKET` rows.
- Each priced card should show full home/draw/away model probabilities plus the selected headline recommendation.
- Keep scheduled delivery change-only if needed, but manual previews should remain complete.

## Dixon-Coles rho Calibration (2026-06-21)

### Problem

The production `model_params.json` had no `dixon_coles_rho` field — it was
generated by an older version of `calibrate_model.py` whose grid only contained
`[0.0]`. The Dixon-Coles adjustment was silently disabled, making the model
functionally identical to plain independent Poisson.

### Fix

Expanded the rho grid in `calibrate_model.py`:
```python
# Old (never searched rho):
dixon_coles_rho_grid = [-0.10, -0.05, 0.0, 0.05]

# New:
dixon_coles_rho_grid = [-0.30, -0.20, -0.15, -0.10, -0.05, 0.0]
```

Reran calibration with `--start-year 2022 --end-year 2026 --tournaments "WC,WQT,F" --rolling`.

### Results

| Parameter | Before | After |
|-----------|--------|-------|
| base_goals | 1.05 | 1.25 |
| elo_divisor | 950 | 1100 |
| home_advantage_elo | 75 | 75 |
| dixon_coles_rho | 0.0 (absent) | **-0.30** |

| Metric | Before | After | Change |
|--------|--------|-------|--------|
| test_log_loss | 0.8041 | 0.7797 | **-3.0%** |
| test_brier | 0.4615 | 0.4505 | **-2.4%** |
| rolling_test_ll | 0.8343 | 0.8286 | -0.7% |
| rolling_test_brier | 0.4864 | 0.4845 | -0.4% |

rho=-0.30 hit the grid boundary, suggesting the true optimum may be more
negative. However, -0.30 is at the lower end of Dixon-Coles literature values.

### Notes

- Calibration grid grew from 180 → 1080 parameter combinations; rolling
  backtest with 65 folds takes ~3 minutes on M-series Mac.

## Per-Tournament Calibration + Tournament-Code Bug Fix (2026-06-22)

### Tournament-Code Bug (WQT → WQ)

The P0 calibration command used `--tournaments "WC,WQT,F"`, but eloratings.net
uses `WQ` for World Cup Qualifiers — **not** `WQT`. The `WQT` code does not
exist in any year's TSV data. This silently dropped **872 qualifier matches**
(~38% of available data).

**Fix:** Changed default to `--tournaments "WC,WQ,F"` and raised `--limit`
from 1500 to 2500 to accommodate the larger dataset.

**Always verify tournament codes** by inspecting raw TSV `fields[7]` values:

```python
from collections import Counter
for line in text.splitlines()[1:]:
    fields = line.split("\t")
    if len(fields) >= 12:
        Counter[fields[7].strip()] += 1
```

Key eloratings tournament codes:

| Code | Meaning | Typical annual volume |
|------|---------|----------------------|
| `WC` | World Cup (finals) | 0-64 |
| `WQ` | World Cup Qualifiers | 82-491 |
| `F` | Friendlies | 207-344 |
| `FQ` | Friendly Qualifiers | 54-151 |
| `WQS` | WC Qualifier (sub-regional) | 56-71 |

### Per-Tournament Calibration

Added `--by-tournament` flag to `calibrate_model.py` and `tournament` field to
`HistoricalMatch` dataclass. Each tournament type gets its own parameter set
in `model_params.json["by_tournament"]`.

**Code changes (7 files):**
- `calibration.py` — `HistoricalMatch` gains `tournament: str = ""` field
- `calibrate_model.py` — `--by-tournament` flag, groups matches by tournament code
- `model_config.py` — `ModelConfig.by_tournament` dict + `params_for(tournament)` method
- `report_cli.py` — `_scan_matches` and `_format_simulation_report` call `model_config.params_for(match.tournament)`
- `reports.py` — `MatchScheduleItem` gains `tournament: str | None`
- `schedule.json` — all matches tagged `tournament="WC"`
- Tests updated for new `tournament` field

**Write-order pitfall:** The `--by-tournament` block must execute BEFORE
`output.write_text()`. If placed after, the `payload["by_tournament"]` dict is
populated but never persisted to disk — the JSON will have `"by_tournament": {}`.

## Reference Project Survey (2026-06-21)

Survey of popular open-source football match prediction approaches for
model improvement directions:

| Technique | Source | Priority | Status in current model |
|-----------|--------|----------|------------------------|
| Dixon-Coles rho calibration | anguswilliams91/world_cup_predictions | P0 | ✅ Done (2026-06-21) |
| Tournament-code fix + WQ data | eloratings.net TSV inspection | P1 | ✅ Done (2026-06-22) — WQT→WQ, +872 matches |
| Per-tournament calibration | — | P1 | ✅ Done (2026-06-22) — WC/WQ/F separate params |
| Opponent-quality-weighted form | SPI attack/defense decomposition | P1 | Form uses raw goals, not opponent-weighted |
| Bayesian parameter estimation | academic PyMC/Stan repos | P1 | Currently grid search only |
| Time-varying Elo (state-space) | academic hierarchical models | P1 | Uses static Elo snapshot |
| SPI dual-dimension ratings | FiveThirtyEight | P2 | Not implemented (Elo + form multipliers instead) |
| Squad-level features (market value, age) | Transfermarkt-based repos | P2 | Not implemented |
| Tournament stage effects | Various | P2 | Not implemented |

### Key takeaway from survey

The most effective open-source WC predictors combine Dixon-Coles with
Bayesian methods and Elo priors. Biggest marginal gains over current model:
1. Proper Bayesian uncertainty (PyMC posterior vs point estimate)
2. Player/squad-level features
3. Time-varying team strength
